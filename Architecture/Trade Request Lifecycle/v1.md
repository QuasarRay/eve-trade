# Trade Request Lifecycle

## Status

Current for the Encore runtime architecture.

## Happy Path

1. The game frontend or simulator signs an `eve-trade-edge.v1` UDP envelope
   containing an `eve-trade-gui.v1` GUI payload.
2. Quilkin forwards the UDP packet to the Encore gateway listener.
3. The gateway enforces packet size, empty-packet rejection, HMAC integrity,
   principal binding, rate limiting, replay protection, bounded queueing, and
   downstream timeout.
4. The gateway calls the private Encore Market API directly with the raw GUI
   payload.
5. Market decodes the GUI interaction, maps the action to issue, accept, or
   cancel, validates current trade/item/wallet snapshots, and builds explicit
   settlement operations.
6. Market publishes typed settlement work to `settlement.WorkTopic` and returns
   a compact queued response to the gateway.
7. The settlementworker subscription receives the work at least once, calls Rust
   `trade-settlement` over standard gRPC, and publishes a typed result message.
8. Rust `trade-settlement` performs the atomic PostgreSQL transaction and owns
   durable settlement state.

## Failure Handling

Gateway failures return compact stable UDP errors and do not expose framework or
payload internals. Market validation failures are returned through Encore errors
and mapped by the gateway. Settlement execution after publication is
asynchronous; correctness depends on stable idempotency keys, request
fingerprints, durable Rust-side idempotency records, and at-least-once Pub/Sub
delivery.

## Boundary Rule

Market owns game-trade interpretation. Settlementworker only converts typed
settlement work to the Rust protobuf request and invokes the Rust service.
