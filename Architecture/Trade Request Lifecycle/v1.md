# Trade Request Lifecycle

## Status

Current for the Encore runtime architecture.

## Happy Path

1. The game frontend or simulator signs a domain-separated `eve-trade-edge.v2` UDP envelope
   containing an `eve-trade-gui.v1` GUI payload.
2. Quilkin forwards the UDP packet to the Encore gateway listener.
3. The gateway enforces packet size, empty-packet rejection, HMAC integrity,
   principal binding, rate limiting, replay protection, bounded queueing, and
   downstream timeout.
4. The gateway calls the private Encore Market API directly with the raw GUI
   payload.
5. Market decodes the GUI interaction, maps the action to issue, accept, or
   cancel, validates current trade/item/wallet snapshots, and builds explicit
   settlement operations.
6. Market asks Rust to durably create a `settlement_operation`, publishes typed
   settlement work, and returns its stable operation ID, queue time, and `queued` state.
7. The settlementworker moves the operation through `processing`, executes the
   typed `ISSUE`, `ACCEPT`, or `CANCEL` request, and publishes a durable result.
8. Rust independently validates parent-intent grammar and actor/resource authority
   before performing the atomic PostgreSQL transaction.
9. Market's idempotent result projection records terminal `succeeded` or `failed`
   state. Clients query `/market/operations/:operation_id` for the durable outcome.

## Failure Handling

Gateway failures return compact stable UDP errors and do not expose framework or
payload internals. Market validation failures are returned through Encore errors
and mapped by the gateway. Settlement execution after publication is
asynchronous. Work redelivery after commit reads the durable operation and retries
only pending result publication; duplicate work and duplicate results are harmless.
Market request fingerprints and canonical Rust execution fingerprints are separate
domains and are compared exactly within their owning layer.

## Replay And Signing

The v2 request and response HMACs bind a domain separator, schema version,
algorithm, key ID, and canonical UTF-8 JSON payload. Gateway replay suppression is
bounded and replica-local. Durable Market and Rust idempotency are the cross-replica
correctness boundary.

## Boundary Rule

Market owns game-trade interpretation. Settlementworker only converts typed work,
coordinates durable lifecycle state, and invokes Rust. Rust is the sole database
writer and does not trust a locally valid Go plan without its own grammar and actor
authorization checks.
