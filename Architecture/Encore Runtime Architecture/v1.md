# Encore Runtime Architecture

## Runtime Flow

```text
game frontend -> Quilkin UDP -> Encore gateway -> Market -> Encore Pub/Sub settlement work -> settlement worker -> Rust trade-settlement -> PostgreSQL
```

The Go backend is one Encore application rooted at the repository root. `gateway`, `market`, and `settlementworker` are Encore service packages. Reusable game-trade rules live in `internal/gametrade`; observability helpers live in `internal/observability`; typed settlement messages live in `settlement`.

## Boundaries

- Quilkin and the UDP packet format remain the external game edge.
- The gateway keeps only UDP-specific validation, HMAC, replay, rate-limit, bounded queue, worker, timeout, and compact response behavior.
- Gateway-to-Market is a private type-safe Encore API call.
- Market owns game-trade interpretation, validation, idempotency keys, request fingerprints, and settlement operation construction.
- Market publishes typed settlement work to Encore Pub/Sub.
- The settlement worker owns the Pub/Sub subscription and invokes the Rust `trade-settlement` service through the remaining cross-language gRPC contract.
- Rust `trade-settlement` owns PostgreSQL mutation, settlement idempotency, and transaction correctness.

## Deployment Shape

Kubernetes deploys one Encore backend workload for the Go services, plus Quilkin, NSQ for self-hosted Pub/Sub, Rust `trade-settlement`, PostgreSQL dependencies, observability, and Litmus resources. Terraform image variables now refer to the Encore backend image and the remaining non-Go images.

Local Go development uses `encore run`. Deployable Go images are produced with:

```powershell
encore build docker --config infra/encore/self-host.nsq.json eve-trade/encore-backend:<tag>
```

Non-Go components still use their own packaging where required.
